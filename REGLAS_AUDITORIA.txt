🏛️ CONFIGURACIÓN DE REGLAS - MOTOR EXPERTO REVISOR 2.0

Este documento detalla qué aspectos el sistema considera OBLIGATORIOS (Errores Críticos) 
y cuáles considera SUGERENCIAS (Advertencias).

===============================================================================
🔴 1. REGLAS CRÍTICAS (OBLIGATORIAS / ERRORES)
===============================================================================
Si alguno de estos puntos falla, el sistema restará puntos significativos y lo marcará en ROJO.

A. CONFIGURACIÓN DE PÁGINA:
   - Tamaño de Papel: Debe ser estrictamente A4.
   - Margen Superior: 2.5 cm.
   - Margen Inferior: 2.5 cm.
   - Margen Derecho: 2.5 cm.
   - Margen Izquierdo: 3.0 cm (Para encuadernación).

B. JERARQUÍA DE TÍTULOS (NIVEL 1):
   - Títulos (RESUMEN, INTRODUCCIÓN, CAPÍTULOS, CONCLUSIONES, etc.):
     * Tamaño: 16pt.
     * Estilo: Negrita (Bold).
     * Alineación: Centrado.

C. JERARQUÍA DE TÍTULOS Y SANGRÍAS (NIVELES 2 AL 5):
   - Nivel 2 (ej. 1.1.): 
     * Título: Justificado o Izquierda, Negrita. Sangría Izq: 0cm, Francesa: 1.25cm.
     * Contenido: Justificado. Sangría Izq: 0cm, Primera línea: 1.25cm.
   - Nivel 3 (ej. 1.1.1.): 
     * Título: Justificado, Negrita. Sangría Izq: 1.25cm, Francesa: 1.25cm.
     * Contenido: Justificado. Sangría Izq: 1.25cm, Primera línea: 1.25cm.
   - Niveles 4 y 5: 
     * Título: Justificado, Negrita. Sangría Izq: 2.5cm, Francesa: 1.5cm.
     * Contenido: Justificado. Sangría Izq: 2.5cm, Primera línea: 1.25cm.
   - Sangría de Viñetas (Depende del nivel de contenido):
     * Bajo Nivel 1 y 2: Sangría Izq: 0.5cm, Francesa: 0.75cm.
     * Bajo Nivel 3: Sangría Izq: 1.75cm, Francesa: 0.75cm.
     * Bajo Nivel 4 y 5: Sangría Izq: 3.0cm, Francesa: 0.75cm.

D. CUERPO DEL TEXTO:
   - Alineación: Debe estar Justificado (ambos lados).
   - Interlineado: Debe ser estrictamente 2.0 (Doble).
   - Tablas y Figuras (Formato Superior):
     * Etiqueta (ej. Tabla 1): 12pt, Negrita, Fuera de la tabla, Sangría Izquierda igual al nivel del título.
     * Título Descriptivo: 12pt, Cursiva, SIN Negrita, Fuera de la tabla, Sangría Izquierda igual al nivel del título.
   - Contenido Interno de la Tabla (Formato APA):
     * Encabezado (Primera fila): Debe estar Centrado y en Negrita.
     * Cuerpo de la Tabla: Interlineado de 1.0 o 1.5. Las reglas de texto general (ej. justificado) no aplican aquí.
     * Bordes: Se exige el uso de líneas horizontales (Formato APA).
   - Notas y Fuentes (debajo de Tablas/Figuras):
     * Tamaño: 10pt.
     * Estilo Etiqueta (Nota: / Fuente:): Normal (Sin negrita, SIN CURSIVA).
     * Estilo Contenido: Puede llevar cursiva si es descripción técnica.
     * Puntuación: Debe usar dos puntos (:), nunca punto y coma (;).
     * Espaciado Posterior: 15pt obligatorio para separar del siguiente párrafo.
     * Sangría Izquierda: Igual al nivel del título al que pertenece la tabla/figura.

E. ESTRUCTURA DE CONTENIDO:
   - Índice General: 
     * Tamaño de fuente: 16pt para el título principal, 12pt para el contenido.
     * Alineación CAPÍTULOS y Secciones Finales: Debe estar estrictamente Centrado.
     * Alineación Títulos descriptivos y sub-ítems: Debe estar Justificado (ambos lados).
     * Presencia: Si el documento tiene Dedicatoria/Agradecimientos, estos deben figurar obligatoriamente en el Índice.
   - Palabras Clave: Debe empezar exactamente con el tag "Palabras clave:" (en minúscula, excepto la P).
   - Sangría: Los párrafos deben tener sangría de primera línea de 1.25 cm.
   - Agradecimientos: Debe ocupar máximo una sola página de la tesis.
   - Extensión del Resumen: Debe tener entre 250 y 300 palabras.
   - Documentos Obligatorios: Debe incluir "Declaración Jurada de Autenticidad" y "Autorización para el Depósito" (Anexos UNAP).

===============================================================================
🟡 2. REGLAS DE SUGERENCIA (ADVERTENCIAS)
===============================================================================
Estos puntos se marcan en AMARILLO. No impiden el depósito, pero se recomienda corregir 
para una tesis de alta calidad.

A. HÁBITOS DE FORMATO:
   - Saltos de Página: Se advierte si se usan 3 o más "Enters" seguidos en lugar de usar 
     la función oficial "Insertar Salto de Página".

C. CALIDAD DE ESCRITURA (MICRO-EDICIÓN):
   - Espacios Dobles: Detecta espacios repetidos innecesarios.
   - Puntuación: Detecta espacios antes de puntos o comas.
   - Espaciado: Detecta falta de espacio después de signos de puntuación.
   - Oraciones Largas: Advierte sobre oraciones de más de 50 palabras (difíciles de leer).
   - Inicio de Párrafo: Detecta si un párrafo inicia con minúscula.
   - Puntos Finales: Advierte si un párrafo largo no termina con un punto.

Plan de Implementación: Supabase + Panel Admin + Agente AutoFixer IA (con Ciclo de Seguridad)
Objetivo General
Construir un ecosistema donde cada tesis auditada quede registrada, el admin pueda revisar y confirmar observaciones, y un Agente AutoFixer IA que lea ese feedback, mejore automáticamente el código Python de los auditores, re-audite la misma tesis, y solo acepte la mejora si no rompió nada que estaba bien. Si algo se rompe → revierte y reintenta.

Módulo 4 — Agente AutoFixer IA con Ciclo de Seguridad 🔒
Flujo Completo con Validación de Regresión

INICIO (invocado manualmente o cuando el servidor está activo)
│
├─ 1. Leer de Supabase todas las observaciones con:
│        status IN ('confirmed_error', 'false_positive')
│        AND ai_fix_applied = false
│        AND ai_retry_count < 5  ← máx 5 intentos por observación
│
├─ Agrupar por (thesis_id, category, auditor_file)
│
└─ Para cada grupo de feedback:
    ├─ 2. Identificar el archivo auditor responsable
    │      (ej: 'Índice General' → indice_general.py)
    │
    ├─ 3. Leer el código actual del auditor
    │
    ├─ 4. *** SNAPSHOT DE SEGURIDAD ***
    │      Guardar backup en services/auditors/backups/indice_general_YYYYMMDD_HHmmss.py
    │
    ├─ 5. Construir prompt para Groq con:
    │      - Código actual del auditor
    │      - Ejemplos reales del error (paragraph_text, expected, actual)
    │      - Comentario del admin ("el sistema confundió X porque Y")
    │      - Intentos previos (si hubo reintentos, incluir qué falló antes)
    │      - Regla estricta: "Solo mejora la función responsable. No elimines ni cambies
    │                          lógica de otras reglas. Devuelve SOLO el código completo."
    │
    ├─ 6. Groq devuelve el código mejorado
    │
    ├─ 7. *** VALIDACIÓN SINTÁCTICA ***
    │      compile(new_code)  → si falla, revertir inmediatamente sin tocar el archivo
    │
    ├─ 8. *** ESCRIBIR NUEVO CÓDIGO ***
    │      Reemplazar el archivo auditor con el código mejorado
    │
    ├─ 9. *** RE-AUDITAR LA MISMA TESIS ***
    │      _run_audit_pipeline(thesis.file_path, thesis.filename)
    │      (misma tesis que generó el feedback)
    │
    ├─ 10. *** COMPARACIÓN DE REGRESIÓN (EL CONJUNTO DORADO) ***
    │
    │       Antes de aplicar cualquier cambio se construye el "Conjunto Dorado":
    │       Reglas que el sistema NO debe tocar porque ya están bien:
    │
    │       Conjunto Dorado =
    │         (A) Todas las reglas con status='passed' en la auditoría original
    │             (guardadas en original_results_snapshot de Supabase)
    │         +
    │         (B) Todas las observaciones que el ADMIN marcó como 'false_positive'
    │             (confirmadas por humano como "esto NO era un error")
    │
    │       Tras re-auditar, si CUALQUIER regla del Conjunto Dorado ahora aparece
    │       como 'error' o 'warning' → ES REGRESIÓN → REVERTIR.
    │
    │       ─────────────────────────────────────────────────────────────────
    │       RESULTADOS POSIBLES:
    │
    │       ✅ ÉXITO:
    │          - El error/confusión objetivo fue resuelto
    │          - NINGUNA regla del Conjunto Dorado fue afectada
    │          → Mantener el código nuevo
    │          → ai_fix_status = 'applied_ok'
    │          → Registrar en ai_code_fixes (validation_status='passed')
    │          → Actualizar ai_memory.json con el patrón aprendido
    │
    │       ❌ REGRESIÓN TIPO 1 (rompió un 'passed'):
    │          - Una regla que antes tenía status='passed' ahora falla
    │          → Revertir al backup INMEDIATAMENTE
    │          → regression_details = "Regla X pasaba, ahora falla. No aceptado."
    │          → Incrementar ai_retry_count
    │          → Reintentar con instrucción adicional: "NO toques la lógica de la regla X"
    │
    │       ❌ REGRESIÓN TIPO 2 (rompió un falso positivo confirmado por admin):
    │          - Una observación que el admin confirmó como 'false_positive'
    │            (es decir, el admin dijo "esto estaba bien") ahora vuelve a aparecer como error
    │          → Revertir al backup INMEDIATAMENTE
    │          → regression_details = "El admin confirmó que 'Regla Y' no era error,
    │                                    pero el nuevo código la vuelve a reportar."
    │          → Incrementar ai_retry_count
    │          → Reintentar con instrucción: "La regla Y NO debe ser reportada como error"
    │
    │       🔄 NO SOLUCIONADO (sin regresión pero el error original persiste):
    │          → Revertir al backup
    │          → Registrar con validation_status='failed_not_fixed'
    │          → Incrementar ai_retry_count con más contexto del por qué falló
    │
    └─ Si ai_retry_count >= 5:
           → ai_fix_status = 'manual_review_required'
           → El panel de admin muestra alerta: "Este error necesita corrección manual"

    def _build_golden_set(self, original_results: list, 
                           admin_false_positives: list) -> set:
        """
        Construye el Conjunto Dorado de reglas que NUNCA deben romperse:
        1. Las que tenían status='passed' en la auditoría original.
        2. Las que el admin marcó como 'false_positive' (confirmadas como OK por humano).
        """
        passed_rules = {r['rule'] for r in original_results if r['status'] == 'passed'}
        fp_rules = {obs['rule'] for obs in admin_false_positives}
        return passed_rules | fp_rules
    def _check_regression(self, golden_set: set, new_results: list,
                           target_observation: dict) -> dict:
        """
        Compara el Conjunto Dorado contra los nuevos resultados.
        Retorna: {
            'error_fixed': bool,        # ¿El error objetivo fue corregido?

            'regression_type': str,      # 'passed_broken' | 'false_positive_broken' | None
            'regressions': list          # Reglas del Conjunto Dorado que ahora fallan
        }
        """
        new_failed = {r['rule'] for r in new_results 
                      if r['status'] in ('error', 'warning')}
        
        regressions = list(golden_set & new_failed)
        
        # Verificar si el error objetivo fue corregido
        target_rule = target_observation['rule']

            (r for r in new_results if r['rule'] == target_rule), None
        )

                       new_result_for_rule['status'] == 'passed')
        
        return {
            'error_fixed': error_fixed,
            'regression': len(regressions) > 0,
            'regressions': regressions
        }

🔒 Resumen: Las 4 Reglas de Seguridad del AutoFixer
#	Condición	Acción
✅	El error se corrigió y ninguna regla del Conjunto Dorado fue afectada	Mantener el código mejorado
❌	Rompió una regla que antes tenía passed en la auditoría original	Revertir + reintentar indicando qué no tocar
❌	Volvió a reportar como error una observación que el admin ya confirmó como OK (false_positive)	Revertir + reintentar con esa instrucción explícita
🔄	El error objetivo sigue igual (sin regresiones, pero tampoco se arregló)	Revertir y reintentar con más contexto
Conjunto Dorado = reglas con status='passed' en la auditoría original + observaciones que el admin confirmó como false_positive. Si el código mejorado toca cualquier elemento de este conjunto → REVERTIR SIN EXCEPCIÓN.

Verificación Manual (paso a paso)
Subir tesis → verificar registros en Supabase.
Abrir /admin → confirmar error + escribir comentario.
Pulsar "Ejecutar AutoFixer" → el panel muestra el progreso.
Verificar que el auditor fue modificado (git diff services/auditors/).
Re-auditar la misma tesis → el error original debe estar resuelto.
Verificar que ninguna regla que antes pasó ahora falla.
Si hay regresión → panel muestra "Revertido" y el código del auditor vuelve al estado original.
